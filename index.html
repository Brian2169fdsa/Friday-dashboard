ManageAI Dashboard Requirements

Complete requirements for the unified FRIDAY + Charlie dashboard at friday.manageai.io — covering the full operational interface: build pipeline, Cockpit AI assistant with memory, multi-session terminal with Claude Code access, Charlie intelligence integration, new account intake, and all shared components across both systems.

Purpose & Vision

Overview

This is not a dashboard for one system. It is the operational interface for the entire ManageAI AI workforce — a single window where Brian manages builds, Dan manages customer relationships, and the team manages the platform. FRIDAY and Charlie share this interface and share components. A client that Charlie is aligning is the same client that FRIDAY builds for. A skillset that FRIDAY delivers is the same skillset Charlie monitors for adoption.

The guiding vision: Using the Cockpit must feel like using Claude Code in a chat window — not a chatbot. You have full context, files open in scope, tool execution visible inline, persistent memory across sessions, and every action mapped back to a real artifact in the system. The terminal sits alongside it for when you need direct server access. Together they are Brian's complete operational workbench.
The fluidity requirement: When Brian is working on a skillset through the Cockpit — editing files, running commands, updating configuration — every change must automatically surface on that skillset's dashboard page. No manual refresh. No separate update step. The Cockpit session and the skillset page are the same thing viewed from two angles.

Memory Injection

How Previous Sessions Inform New Sessions

When Brian starts a new Cockpit session with a skillset selected, the system prompt automatically includes:

1
Skillset contextCurrent skillset name, platform, version, last build date, QA score, deployment status
2
Recent session summaries (last 5)The Claude-generated summary of each previous session for this skillset. FRIDAY arrives knowing what was discussed, what was changed, what was left unresolved.
3
Changes madeLast 10 workflow_changes entries for this skillset — what was built, fixed, patched, when, by whom.
4
Open issuesAny friday_issues rows with fix_status='open' for this client — FRIDAY knows what's broken before Brian mentions it.
5
Files last touchedList of files read/written in the most recent session — so FRIDAY knows where to look first.
The result: Brian opens a new session and types "where did we leave off?" — FRIDAY gives a complete answer. Brian types "fix the webhook issue" — FRIDAY already knows which webhook, which workflow, and what was tried last time.
Auto-Summarize

Session Summary Generation

When Brian switches sessions or navigates away, POST /api/cockpit/summarize fires in the background with the full message history. Claude generates a 2–3 sentence summary that captures: what was worked on, what was changed, what was left incomplete. Stored in cockpit_chats.summary. Used as input for the next session's memory injection.

Summary format example: "Fixed the Google Sheets retry logic in WF-01 for Apex Logistics — added 5-minute backoff after quota errors. Blueprint updated and pushed to GitHub. Still need to test the activation webhook — left as open item."

3.3 Context & Skillset Mapping

Cockpit

Every Cockpit session must know what it is working on. Every action taken in a session must map back to the correct skillset and client so the dashboard stays live.

Context Selector

Left Sidebar — Session Controls

  • Client dropdown: Loads all friday_customers. Default: "— General —". On select: loads skillset dropdown, filters session list, injects client context into system prompt.
  • Skillset dropdown: Hidden until client selected. Loads friday_skillsets WHERE customer_id = selected. On select: injects full skillset context + memory.

If Brian opens a build from the Skillset Detail page via [🤖 Build Agent], the Cockpit opens pre-loaded with that client + skillset already selected and memory already injected. Brian never has to select context manually when arriving from a linked button.

4.1 Terminal Overview

Terminal

The Terminal tab provides direct server access with multiple named sessions. Unlike the Cockpit (which uses AI to mediate every action), the Terminal gives raw access — when Brian needs to see exactly what's happening on the server or run Claude Code directly, this is where he goes.

Relationship to Cockpit: The Cockpit handles 90% of server interactions through the run_command and other tools — with AI context, memory, and automatic dashboard updates. The Terminal handles the other 10%: complex interactive sessions, long-running processes, Claude Code autonomous builds, and direct database inspection. Both live in the dashboard and complement each other.
SSL status: terminal.manageai.io SSL cert pending (Let's Encrypt DNS challenge). Until cert issued: each session opens in a new browser tab at https://5.223.79.255:7682. Once cert issued: sessions embed as iframes in the Terminal tab.

Session Persistence

How Sessions Survive Reconnects

Each session is a named tmux window on the server. When Brian closes the browser or navigates away, the tmux session keeps running. When he returns, the session reconnects and he sees the full scrollback history.

  • tmux session name: friday-{sessionType}-{n}
  • Reconnect: ttyd connects to the existing tmux window by name
  • Scrollback: 10,000 lines per session
  • Session list persisted in Supabase: friday_terminal_sessions table (id, name, type, tmux_window, last_active)
